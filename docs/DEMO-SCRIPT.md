# Demo 视频讲解稿（口播 + 操作提示）

> 给你排练用。**左栏「画面操作」** = 你在屏幕上点什么；**右栏「口播台词」** = 你照着念（可口语化微调）。
> 目标时长 **4–5 分钟**。配音用中文。评分主轴要全程暗扣：① 完整度与创新 40% ② 开发过程与质量（架构/代码/PR）40% ③ demo 表达 20%。
>
> ⚠️ 讲稿按 **PR10 完成后的形态**写（多体裁示例选择器 + 打磨后的界面）。这几块我正在并行做；录制前我会告诉你哪些已就位。若录制时某块还没好，用 `[备用]` 段落顶替。

---

## 0. 录制前 checklist（别省）

- [ ] `npm run dev` 起好，浏览器开 `http://localhost:3000`，窗口拉到 1280px 以上、关掉多余标签页。
- [ ] `.env.local` 里 `DEEPSEEK_API_KEY` 配好（真跑要用）。
- [ ] **提前空跑一遍**：选示例→转换跑通一次，确认网络/模型正常（真 DeepSeek 红楼≈30s、散文≈100s，录制时别干等——见下「录制技巧」）。
- [ ] 页面刷新到干净初始态（空输入、引导卡）。
- [ ] 录屏软件 + 麦克风测一句。
- [ ] **录制技巧**：转换要等几十秒。两种拍法选一种——(A) 真录等待，事后剪掉中间，保留时间线点亮的关键帧；(B) 转换前先念「架构讲解」那段，让等待和讲解重叠。**推荐 B**。

## 1. 开场 · 痛点（~30s）

| 画面操作 | 口播台词 |
|---|---|
| 停在干净首页，不动 | 「把一部小说改成影视剧本，是影视工业的第一道工序，但也是最枯燥、最容易出错的一道。**几十万字里，同一个人物在不同章节有不同称呼、场景的内外景和时间需要逐一标注、每句对白要拆出来对应到角色**——人工做又慢又容易前后矛盾。」 |
| | 「我做的这个工具，让作者**粘贴或上传至少三章小说，AI 自动产出一份结构化、可编辑、而且每个场景都能溯源回原文的影视剧本初稿**。下面我演示完整流程。」 |

> 暗扣点：开场就点出「跨章一致性」「结构化」「溯源」三个后面要兑现的承诺。

## 2. 输入 · 多体裁示例 + 多文件上传（~40s）

| 画面操作 | 口播台词 |
|---|---|
| 点开**示例选择器**，让下拉/卡片露出几类体裁 | 「我内置了一组不同体裁的示例——**古典章回体的《红楼梦》、现代网文、还有散文**。每一类都标注了它**重点考验 AI 的哪种能力**：比如散文没有明确的戏剧冲突，考验的是 AI 能不能从写景抒情里**自动提取出可视化的场景**。这几类我都**真实端到端跑通过**，不是摆设。」 |
| 选「古典章回·红楼梦」 | 「我先用《红楼梦》前几回演示——群戏多、对白密、人物别名多，最能体现跨章一致性。」 |
| 鼠标划过「上传 .txt」 | 「如果是自己的稿子，可以直接粘贴，或者**一次选多个 txt 文件**——工具会**按文件名里的章节号自然排序**再拼接，『第 2 章』排在『第 10 章』前面，不会按字典序乱掉。」 |

> 暗扣点：多体裁 = 完整度/创新；「真跑通过」= 诚实、有底气；自然序 = 工程细节。

## 3. 核心 · Agent 流水线时间线（~90s，全片重点）

| 画面操作 | 口播台词 |
|---|---|
| 点「转换」，时间线开始点亮 | 「点转换。注意中间这条**时间线**——它在实时展示一支『AI 剧组』在协作。这套多 agent 架构是整个作品的核心。」 |
| 指「场记」节点 | 「第一步**场记**：一个确定性的工具，把全文切成章和场景候选。它不调大模型，专门识别回目标题、按长度兜底切分、还会检测近重复段落——保证后面喂给模型的每一块都干净、不超长。」 |
| 指「设定集」节点 | 「第二步**设定集**：第一个 AI agent，扫一遍全文，产出统一的**人物表和地点表**。关键在这里——它会把『林黛玉、黛玉、颦儿』这些**别名合并成同一个角色，给一个稳定 ID**。这张设定集就是整个流水线的**共享记忆**，被后面每一个场景引用。这就是跨章一致性的来源。」 |
| 指「场景编剧」节点，看 `x/9` 递增 | 「第三步**场景编剧**：逐个场景把叙事改写成剧本元素流。它**只能引用设定集里已有的角色 ID，不能自己新增人物**——从源头上杜绝 AI 幻觉出一个原著里没有的角色。」 |
| 稍停，讲自纠（趁等待） | 「每个场景出来后，还有两道关：一个**确定性校验**——检查引用的 ID 是否真实存在、格式是否合规；一个 **AI 责编**——做语义自评，挑人物矛盾、漏掉的对白。**任何一关不过，就带着具体错误反馈打回场景编剧重写，最多重试两次**。这就是 agent 的自我纠错闭环。」 |
| 指「导演」节点变✓ | 「最后**导演**把所有场景汇编成完整剧本。整条流水线就是 agent 的四种核心能力：**规划、共享记忆、工具调用、自我纠错**，全在这一条时间线上看得见。」 |

> 暗扣点：这一段直接对应「开发过程与质量 40%」里的架构分。把四能力点名说出来。

## 4. 输出 · 剧本卡片 + YAML Schema 设计（~70s）

| 画面操作 | 口播台词 |
|---|---|
| 右侧卡片流式出现，滚一下 | 「剧本出来了。**卡片视图**给人读：每个场景有内外景、日夜、场景概要，下面是动作和对白交错的元素流，角色名都已经从 ID 解析成了中文名。」 |
| 切到「YAML」源码视图 | 「切到 **YAML 源码**视图——这是作品要求里**唯一明确要自定义 Schema 并书面论证设计的部分**，我重点讲三个设计决策。」 |
| 指 `characters:` 和场景里的 `*_id` | 「第一，**用 ID 引用而不是内联人物名**。场景里写的是 `character_id`，指向顶部的人物表。这样改一处全局生效，机器能校验引用完整性，也正好是 agent 跨章记忆的落点。」 |
| 指某场景 `elements:` 列表 | 「第二，**elements 是有序的异构列表**——动作、对白、转场混在一个有序数组里。因为剧本的本质是**线性时间流**，固定字段表达不了『动作—对白—动作』这种交错。」 |
| 指 `source:` 字段 | 「第三，每个场景都带 **source 溯源字段**，记录它来自原文第几章、哪一段。这是用来对抗模型幻觉、建立作者信任的——下一个功能就靠它。」 |

> 暗扣点：Schema 论证 = 产品架构师核心加分项，慢慢讲、讲透三个「为什么」。

## 5. 价值闭环 · 溯源 + 编辑 + 导出（~60s）

| 画面操作 | 口播台词 |
|---|---|
| 切回卡片，点某场景的「溯源」 | 「点这个场景的**溯源**——它直接跳回原著，把这个场景对应的原文段落**高亮**出来。AI 生成的每一句，作者都能一秒核对它从哪来，不用怕它瞎编。这就是『可信』。」 |
| 关弹层，切 YAML，改一句对白 | 「再看『可编辑』。这份 YAML 是**可以直接改的**。我改一句对白……」 |
| 点「应用」，卡片同步更新 | 「点应用——它会**重新过一遍校验和 schema**，没问题才回灌。卡片视图实时同步了。如果我改出一个不存在的角色 ID，它会**警示我引用断了，但保留住我之前的好状态**，不会把整份剧本搞坏。」 |
| 点「导出」，下载 .yaml | 「最后**导出**成 YAML 文件，交给后续的渲染器或排版工具。整个产品闭环——**生成、溯源、编辑、导出**——就走通了。」 |

> 暗扣点：溯源+编辑+导出 = 完整度与产品价值闭环。

| | 「配套一套**为剧本量身设计、并书面论证过的 YAML Schema**，支撑跨章一致性和引用校验。整个过程**按职责拆成了十多个独立 PR，每个都带测试、先写测试再实现**。它现在能吃多种体裁的真实小说，端到端跑通、可编辑、可溯源。谢谢观看。」 |

> 暗扣点：最后一句把「PR 数量 + commit 分布 + TDD」明说出来，扣满 40% 工程分。

## 备用 / 加分话术（时间够再用）

- **吃脏输入**：「它对脏输入也鲁棒——没有章回的散文会按长度自动兜底切分，重复段落会被标成『待人工复核』而不是悄悄删掉。」
- **模型可换**：「LLM 走的是 OpenAI 兼容抽象层，baseURL、key、模型全可配，演示用的是 DeepSeek，但代码不绑定任何一家。」
- **一句价值**：「它把一个资深编剧助理几天的体力活，压缩到几十秒，且每一句都可回查。」

## 录制顺序建议

1. 先把 §3 那段「架构讲解」练熟——它最长、最重要，也用来填转换等待时间。
2. §4 的 Schema 三点，每点配一次屏幕指向，别赶。
3. 全程语速放慢，点一下、停一下、再说。宁可 5 分钟讲清楚，不要 3 分钟糊过去。
